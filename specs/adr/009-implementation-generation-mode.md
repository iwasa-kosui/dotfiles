# ADR-009: 実装生成モードの追加

## ステータス

承認（ADR-001 を拡張）

## コンテキスト

当初の4モード構成（分析/仕様作成/レビュー/更新）は、仕様書の生成までをスコープとしていた。しかし2025年にSpec-Driven Developmentが確立し「Specify → Plan → Tasks → Implement」のワークフローが標準化された。仕様書からタスク分解・コード生成・受け入れテスト生成までをスコープに含めることで、RDRA要件から実装までの一貫したトレーサビリティを実現できる。

選択肢:
1. **仕様作成モードまでに留める**: 従来どおり文書生成のみ
2. **実装生成モードを追加**: 仕様→タスク→コード/ADR/テスト生成を5番目のモードとして追加

## 決定

**実装生成モードを5番目のモードとして追加**し、5モード構成とする。

## 理由

- **SDDワークフローの完結**: RDRA要件定義からコード生成まで一貫して実行できる。GitHub Spec Kitの「Specify → Plan → Tasks → Implement」ワークフローをRDRAトレーサビリティ付きで実現する
- **トレーサビリティの延伸**: `traces_to` チェーンがGOAL → REQ → BUC → UC → FR → SPEC → TASK → CODE まで到達し、任意のコードから「なぜこの実装が必要か」をRDRA要件まで遡れる
- **先行ツールとの差別化**: RDRAAgentは仕様たたき台まで、GitHub Spec KitはRDRA接続なし。両者のギャップを埋めるポジションを取れる
- **受け入れテスト生成**: RDRA BUC/UCから受け入れテストを生成することで、要件ベースのテスト戦略を自動化できる

## トレードオフ

- スキルの複雑性が増す → 実装生成モードの詳細は `references/` に委譲し、SKILL.md本体は500行以下を維持
- コード生成の品質はLLMの能力に依存 → 生成コードは必ず人間がレビューする前提とし、スキルはあくまで「たたき台」を提供する位置づけ
- 技術スタック依存 → 実装生成モード開始時にユーザーから技術スタック・アーキテクチャ方針をヒアリングする
