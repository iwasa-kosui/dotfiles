# ADR-003: バーティカルスライス + 共有モデルのディレクトリ構造

## ステータス

承認

## コンテキスト

RDRA成果物のディレクトリ構造をどう設計するか。RDRA 3.0で「コンテキスト」が正式導入され、情報・状態・条件・バリエーションをビジネス観点でグルーピングする概念が標準化された。選択肢は:

1. **レイヤー別**: `value/`, `environment/`, `boundary/`, `system/` で分割
2. **バーティカルスライス**: RDRA 3.0のコンテキスト単位で分割し、各ファイルが4レイヤーを縦貫
3. **フラット**: 全ファイルを1ディレクトリに配置

## 決定

**RDRA 3.0のコンテキスト単位によるバーティカルスライス + 共有モデル**の構造を採用する。

```
rdra/
├── overview.md              # 全体概観
├── contexts/                # ビジネスコンテキスト別
│   ├── {context-name}.md    # 4レイヤー縦貫
│   └── ...
├── shared/                  # コンテキスト横断
│   ├── information-model.md
│   └── state-models.md
└── traceability.yaml
```

## 理由

- **RDRA 3.0コンテキストとの整合**: RDRA 3.0で正式導入されたコンテキスト（情報・状態・条件・バリエーションのビジネス観点グルーピング）をそのままディレクトリ構造に反映できる
- **レイヤー分断の回避**: レイヤー別構造ではビジネスユースケースの全体像を把握するために複数ディレクトリを行き来する必要がある。バーティカルスライスなら1ファイルで「なぜ必要か → どう使われるか → 何と接するか → 何を扱うか」を通して読める
- **overview.mdで俯瞰**: 個別コンテキストに潜らなくても、アクター一覧・ゴール一覧・コンテキスト間の関係が見える
- **共有モデルの重複防止**: 情報モデル（エンティティ）や状態モデルは複数のコンテキストから参照される。`shared/` に分離することでDRYを保つ。各コンテキストからは `INFO-*` / `STATE-*` IDで参照する
- **スケーラビリティ**: コンテキストが増えてもファイルを追加するだけ。レイヤー別だと1ファイル内のセクションが肥大化する

## トレードオフ

- 1ファイルが長くなる可能性 → 各セクション（価値/環境/境界/システム）に明確な区切りを設け、目次を付与
- 共有モデルへの参照が暗黙的になる → `traceability.yaml` で明示的にリンクし、レビューモードで未参照チェック
