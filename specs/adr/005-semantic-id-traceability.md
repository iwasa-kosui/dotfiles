# ADR-005: 要素種別ベースID + traces_to によるトレーサビリティ

## ステータス

承認

## コンテキスト

RDRA要素の識別方法とトレーサビリティの実現方式を決める必要がある。選択肢は:

1. **レイヤー略称ベースID**: `V-001`（価値）, `E-001`（環境）, `B-001`（境界）, `S-001`（システム）
2. **要素種別ベースID**: `ACTOR-001`, `BUC-001`, `UC-001`, `INFO-001`
3. **自動採番（連番）**: `001`, `002`, ...

## 決定

**要素種別ベースID** + **`traces_to` フィールドによるWhy依存チェーン**を採用する。

### IDスキーマ

| レイヤー | プレフィックス | 対象 |
|---------|---------------|------|
| システム価値 | `ACTOR-*` | アクター（人・外部システム） |
| | `GOAL-*` | システム化の目的・ゴール |
| | `REQ-*` | 要求 |
| 外部環境 | `BIZ-*` | ビジネスコンテキスト（業務） |
| | `BUC-*` | ビジネスユースケース |
| | `VAR-*` | バリエーション・条件 |
| システム境界 | `UC-*` | ユースケース |
| | `SCR-*` | 画面 |
| | `EVT-*` | イベント |
| システム | `INFO-*` | 情報（エンティティ） |
| | `STATE-*` | 状態モデル |

### トレーサビリティ

各要素が `traces_to` フィールドで上位要素を参照し、Why依存チェーンを形成する:

```
GOAL-001 ← REQ-001 ← BUC-001 ← UC-001 ← INFO-001
```

## 理由

- **セマンティックな可読性**: `UC-003` を見ただけで「ユースケース」だとわかる。`B-003` では「境界レイヤーの何か」としかわからない。チーム内のコミュニケーションで「UC-003を確認して」と言える
- **レイヤー内の区別**: 同じシステム境界レイヤーにUC・画面・イベントがあるが、`B-001` では種別不明。`UC-001` / `SCR-001` / `EVT-001` なら一意に識別できる
- **Why依存チェーン**: `traces_to` による上位参照で「なぜこの仕様が必要か」をゴールまで遡れる。レビューモードでチェーンの断絶を検出し、根拠のない仕様を炙り出せる
- **カバレッジチェック**: `traceability.yaml` で全要素の `traces_to` を集約し、「どのゴールから到達できないUCがあるか」等の網羅性チェックが可能

## トレードオフ

- プレフィックスが長い（`ACTOR-001` vs `A-001`） → 可読性を優先。短縮形は曖昧さを生む
- 要素種別が増えた場合にプレフィックスの追加が必要 → RDRAの要素種別は安定しており、頻繁な追加は想定しない
