# ADR-007: PRD/ADR/Spec 3文書構成

## ステータス

承認

## コンテキスト

仕様作成モードで生成するSDD文書の構成をどうするか。選択肢は:

1. **単一仕様書**: すべてを1ファイルにまとめる
2. **PRD + Spec の2文書**: 要件と仕様を分離
3. **PRD + ADR + Spec の3文書**: 要件・判断・仕様を分離

## 決定

**PRD / ADR / Spec の3文書構成**を採用する。

| 文書 | 主な読者 | 内容 |
|------|---------|------|
| PRD | ステークホルダー、PdM | なぜ作るか、誰のために、何を |
| ADR | 将来の開発者、アーキテクト | 技術的判断とその理由 |
| Spec | 実装者、レビュワー | 実装可能な技術仕様 |

## 理由

- **読者層の違い**: PRDはビジネスサイドも読む。ADRはアーキテクチャ判断の記録であり技術者向け。Specは実装者が参照する。読者が違えばドキュメントも分ける
- **ライフサイクルの違い**: PRDはプロジェクト初期に固まりやすい。ADRは判断のたびに追加される。Specは実装と並行して詳細化される。更新頻度の異なる文書を分離することで変更管理が容易
- **ADRの蓄積価値**: 技術的判断を個別のADRファイルとして記録することで、「なぜこの設計になったか」を後から検索・参照できる。単一仕様書に埋め込むと判断の経緯が埋没する
- **トレーサビリティ**: RDRA要件（Why）→ PRD（What）→ ADR（How decided）→ Spec（How implemented）の依存チェーンが明確になる

## トレードオフ

- 文書間の重複リスク → 各文書の責務を明確にし、重複する場合はクロスリファレンスで対応
- 小規模プロジェクトではオーバーキル → 仕様作成モードでスコープに応じてADRを省略可能にする
