# PRD: RDRA-SDD スキル

## 概要

RDRA 2.0に基づく要件定義から仕様駆動開発（SDD）までを一貫して支援するClaude Codeスキル。
チーム開発プロジェクトにおいて、「要件の構造化 → 仕様書生成 → レビュー → 更新」のワークフローをシームレスに繋ぐ。

## 背景と課題

### RDRAのツールサポート不足

RDRA 2.0は構造化された要件定義手法だが、専用ツールはRDRA Modeler（有料）に限定される。多くのチームはExcelやWikiで代替しており、モデル間の整合性チェックや図の生成が手作業になっている。

### 要件定義の属人化

RDRAの4レイヤー（システム価値・外部環境・システム境界・システム）を正しく使い分けるには経験が必要。レイヤー間の関係性を意識しながら要件を構造化できる人材がチームにいないと、断片的な要件リストに終わる。

### SDDへの接続の弱さ

要件定義とソフトウェア仕様書の間にギャップがある。RDRA成果物からPRD/ADR/Specへの変換は手動で行われ、トレーサビリティが失われやすい。「なぜこの仕様になったか」をRDRA要件まで遡って確認できない。

## ユーザーストーリー

### US-1: RDRA分析

> チームリーダーとして、プロジェクトのビジネス要件をRDRA 2.0の4レイヤーで構造化したい。
> 対話的にヒアリングされることで、暗黙知を漏れなく形式知化できるようにしたい。

### US-2: 仕様書作成

> テックリードとして、RDRA成果物からPRD/ADR/Specの3文書を生成したい。
> 要件から仕様への変換を自動化し、トレーサビリティを保ったままSDD成果物を得たい。

### US-3: 仕様レビュー

> プロジェクトマネージャーとして、作成された仕様書をRDRA要件と突合してレビューしたい。
> 網羅性の抜け漏れや要件との不整合を機械的に検出し、レビューの質を均一化したい。

### US-4: 仕様更新

> 開発者として、要件変更が発生した際に影響範囲を把握し、関連する仕様書を更新したい。
> 変更の波及を追跡し、更新漏れを防ぎたい。

### US-5: トレーサビリティ確認

> QAエンジニアとして、任意の仕様項目からRDRA要件の根拠まで遡って確認したい。
> テストケースの妥当性をビジネス要件と紐づけて説明できるようにしたい。

## 機能要件

### 4モード構成

| モード | 目的 | 入力 | 出力 |
|--------|------|------|------|
| **分析** (RDRA Analysis) | ビジネス要件をRDRA 4レイヤーで構造化 | 対話（ヒアリング） | RDRA成果物（YAML + Mermaid図） |
| **仕様作成** (Spec Writing) | RDRA成果物からSDD文書を生成 | RDRA成果物 | PRD / ADR / Spec |
| **レビュー** (Spec Review) | 仕様書の品質をRDRA要件と突合して検証 | RDRA成果物 + 仕様書 | レビューレポート |
| **更新** (Spec Update) | 要件変更に伴う仕様書の更新 | 変更内容 + 既存成果物 | 更新済み成果物 + 変更ログ |

### 入力→出力マトリックス

```
分析モード:
  対話（ヒアリング）
    → overview.md（全体概観）
    → contexts/{name}.md（バーティカルスライス × N）
    → shared/information-model.md
    → shared/state-models.md
    → traceability.yaml

仕様作成モード:
  RDRA成果物一式
    → prd.md
    → adr/001-*.md ... adr/NNN-*.md
    → spec.md
    → traceability.yaml（更新）

レビューモード:
  RDRA成果物 + 仕様書
    → review-report.md（4観点の検証結果）

更新モード:
  変更内容の記述 + 既存成果物一式
    → 更新済みファイル群（RDRA成果物 + SDD文書 + traceability.yaml）
    → change-log.md（影響分析と変更履歴）
```

## 非機能要件

| 項目 | 要件 |
|------|------|
| SKILL.mdサイズ | 500行以下（参照ファイルへの委譲で制御） |
| 対話言語 | 日本語 |
| 図表形式 | Mermaid（GitHubプレビュー互換） |
| 構造データ形式 | YAML（SSoT）/ Mermaid（ビュー） |
| プラットフォーム | Claude Code（CLI / Neovim統合） |

## スコープ

### MVP（今回のスコープ）

- 4モードの基本フロー
- RDRA 4レイヤーの全要素サポート
- バーティカルスライスによるコンテキスト単位の分析
- Mermaid図テンプレート（7種）
- YAMLベースのトレーサビリティ
- SDD 3文書テンプレート（PRD/ADR/Spec）

### 将来拡張

- 既存コードベースからのRDRAリバースエンジニアリング
- CI連携（トレーサビリティカバレッジチェック）
- 複数プロジェクト間でのRDRAモデル共有
- PlantUML / draw.io 出力対応
- RDRA Modeler形式のインポート/エクスポート

## 成功指標

1. **構造化の完全性**: 分析モードで生成されたRDRA成果物が4レイヤーを網羅していること
2. **トレーサビリティの連鎖**: 任意の仕様項目からRDRA要件まで `traces_to` チェーンで遡れること
3. **レビュー検出率**: レビューモードが網羅性・整合性の問題を検出できること
4. **更新の波及追跡**: 更新モードが変更影響を正しく特定し、更新漏れがないこと
5. **GitHub互換性**: 生成されたMermaid図がGitHubのMarkdownプレビューで正常表示されること
