# PRD: RDRA-SDD スキル

## 概要

RDRA 3.0に基づく要件定義から仕様駆動開発（SDD）までを一貫して支援するClaude Codeスキル。
「要件の構造化 → 仕様書生成 → タスク分解 → コード・ADR・受け入れテスト生成 → レビュー → 更新」のワークフローをシームレスに繋ぐ。

## 背景と課題

### RDRAのツール分散

RDRA 3.0は表形式の定義とRDRAGraphによる可視化をスプレッドシート上で実現しているが、スプレッドシートからSDD成果物への接続は手動のままである。RDRAAgent（神崎善司氏公開）はClaude Codeを用いた4フェーズの要件定義を提供するが、仕様書生成・タスク分解・コード生成までは含まない。Enterprise Architectの無料RDRAアドインやRDRA定義・分析Sheetなど複数のツールが存在するが、いずれもRDRA成果物の生成に留まる。

### 要件定義の属人化

RDRAの4レイヤー（システム価値・外部環境・システム境界・システム）を正しく使い分けるには経験が必要。RDRA 3.0で表形式に簡略化されたが、レイヤー間の関係性を意識しながら要件を構造化できる人材がチームにいないと、断片的な要件リストに終わる。

### SDDワークフローの断絶

2025年にSpec-Driven Developmentが確立し「Specify → Plan → Tasks → Implement」のワークフローが標準化された（GitHub Spec Kit, Kiro, OpenSpec等）。しかしRDRA成果物からこのSDDワークフローへの入力を生成するツールは存在しない。要件定義と仕様駆動開発の間にギャップがある。

## ユーザーストーリー

### US-1: RDRA分析

> チームリーダーとして、プロジェクトのビジネス要件をRDRA 3.0の4レイヤーで構造化したい。
> 対話的にヒアリングされることで、暗黙知を漏れなく形式知化できるようにしたい。

### US-2: 仕様書作成

> テックリードとして、RDRA成果物からPRD/ADR/Specの3文書を生成したい。
> 要件から仕様への変換を自動化し、トレーサビリティを保ったままSDD成果物を得たい。

### US-3: タスク分解・実装生成

> 開発者として、仕様書からタスクリスト・実装コード・ADR・受け入れテストを生成したい。
> SDDの「Specify → Plan → Tasks → Implement」ワークフローをRDRA要件に紐づけて実行したい。

### US-4: 仕様レビュー

> プロジェクトマネージャーとして、作成された仕様書をRDRA要件と突合してレビューしたい。
> 網羅性の抜け漏れや要件との不整合を機械的に検出し、レビューの質を均一化したい。

### US-5: 仕様更新

> 開発者として、要件変更が発生した際に影響範囲を把握し、関連する仕様書を更新したい。
> 変更の波及を追跡し、更新漏れを防ぎたい。

### US-6: トレーサビリティ確認

> QAエンジニアとして、任意の仕様項目からRDRA要件の根拠まで遡って確認したい。
> テストケースの妥当性をビジネス要件と紐づけて説明できるようにしたい。

## 機能要件

### 5モード構成

| モード | 目的 | 入力 | 出力 |
|--------|------|------|------|
| **分析** | ビジネス要件をRDRA 3.0の4レイヤーで構造化 | 対話（ヒアリング） | RDRA成果物（YAML + Mermaid図） |
| **仕様作成** | RDRA成果物からSDD文書を生成 | RDRA成果物 | PRD / ADR / Spec |
| **実装生成** | 仕様からタスク・コード・ADR・テストを生成 | RDRA成果物 + 仕様書 | タスクリスト / コード / ADR / 受け入れテスト |
| **レビュー** | 仕様書の品質をRDRA要件と突合して検証 | RDRA成果物 + 仕様書 | レビューレポート |
| **更新** | 要件変更に伴う仕様書の更新 | 変更内容 + 既存成果物 | 更新済み成果物 + 変更ログ |

### 入力→出力マトリックス

```
分析モード:
  対話（ヒアリング）
    → overview.md（全体概観）
    → contexts/{name}.md（バーティカルスライス × N）
    → shared/information-model.md
    → shared/state-models.md
    → traceability.yaml

仕様作成モード:
  RDRA成果物一式
    → prd.md
    → adr/001-*.md ... adr/NNN-*.md
    → spec.md
    → traceability.yaml（更新）

実装生成モード:
  RDRA成果物 + 仕様書
    → tasks.md（タスクリスト）
    → src/**（実装コード）
    → adr/NNN-*.md（技術判断ADR）
    → tests/**（受け入れテスト）
    → traceability.yaml（更新）

レビューモード:
  RDRA成果物 + 仕様書
    → review-report.md（4観点の検証結果）

更新モード:
  変更内容の記述 + 既存成果物一式
    → 更新済みファイル群（RDRA成果物 + SDD文書 + traceability.yaml）
    → change-log.md（影響分析と変更履歴）
```

## 非機能要件

| 項目 | 要件 |
|------|------|
| SKILL.mdサイズ | 500行以下（参照ファイルへの委譲で制御） |
| 対話言語 | 日本語 |
| 図表形式 | Mermaid（GitHubプレビュー互換） |
| 構造データ形式 | YAML（SSoT）/ Mermaid（ビュー） |
| プラットフォーム | Claude Code（CLI / Neovim統合） |

## スコープ

### MVP（今回のスコープ）

- 5モードの基本フロー
- RDRA 3.0の4レイヤー全要素サポート（条件・バリエーションはシステムレイヤー配置）
- バーティカルスライスによるコンテキスト単位の分析
- Mermaid図テンプレート（7種）
- `traces_to` による単方向トレーサビリティ
- SDD 3文書テンプレート（PRD/ADR/Spec）
- 仕様からのタスク分解・コード生成・ADR生成・受け入れテスト生成

### 将来拡張

- 既存コードベースからのRDRAリバースエンジニアリング
- CI連携（トレーサビリティカバレッジチェック）
- 複数プロジェクト間でのRDRAモデル共有
- PlantUML / draw.io 出力対応
- RDRA定義・分析Sheet / RDRAGraph との相互運用
- GitHub Spec Kit / OpenSpec形式でのタスク出力

## 受け入れ条件

1. **分析モード**: RDRA 3.0の4レイヤー要素（ゴール・要求・BUC・UC・画面・イベント・情報・状態・条件・バリエーション）を含むYAML成果物が生成され、パースエラーなく読み込めること
2. **トレーサビリティ**: 任意の仕様項目から `traces_to` チェーンでGOALに到達できること。孤立要素がゼロであること
3. **仕様作成モード**: RDRA成果物からPRD/ADR/Specが生成され、各文書内のRDRA要素ID参照が実在する要素を指していること
4. **実装生成モード**: 仕様書からタスクリストが生成され、各タスクが対応するUCまたはSpec項目への `traces_to` を持つこと。受け入れテストがRDRA要件のBUC/UCに対応していること
5. **レビューモード**: 意図的に欠落・不整合を含むサンプル成果物に対して、網羅性・整合性の問題を検出できること
6. **更新モード**: RDRA要素の変更時に影響を受けるSDD文書が特定され、更新後にトレーサビリティの断絶がないこと
7. **Mermaid互換性**: 生成されたMermaid図がGitHub Markdownプレビューで正常表示されること

## 先行ツールとの差別化

| ツール | 範囲 | 本スキルとの違い |
|--------|------|-----------------|
| RDRAAgent v0.6 | RDRA要件定義 → 仕様たたき台 | SDD文書テンプレート・タスク分解・コード生成・受け入れテスト生成・レビュー・更新モードを含まない |
| RDRA定義・分析Sheet | RDRA要素の表形式入力 | 手動入力のみ。対話的ヒアリングやSDD接続なし |
| RDRAGraph | RDRA要素の可視化 | 可視化のみ。仕様生成や実装生成なし |
| GitHub Spec Kit | SDD（Specify/Plan/Tasks） | RDRA要件定義との接続なし |
| Enterprise Architect | UMLモデリング + RDRA | デスクトップアプリ。Claude Codeとの統合なし |
