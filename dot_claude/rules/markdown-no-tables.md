# Markdownドキュメントでは表形式を避ける

PRD、ADR、README、設計書、ルールなどのMarkdownドキュメントを新規作成・編集する際は、原則として表（`| col | col |` 構文）を使わない。地の文・箇条書き・見出しで表現する。

## 理由

- 表はセル内に長文を書きづらく、内容を無理に短縮する圧力が働く。結果として含意や条件が失われる
- 列ごとの情報量が偏ると視覚的に歪み、可読性がむしろ下がる
- 行の追加・削除でdiffが大きくなり、レビューが難しくなる
- ターミナルや狭い画面では折り返しで崩れて読めない
- 箇条書きや見出しのほうが、ネスト・コードブロック・リンクなどMarkdownの他の要素と組み合わせやすい

## 代替表現

2〜3項目の対比なら**地の文**で十分:

> 開発環境ではモックを使い、本番環境では実APIを呼ぶ。

複数項目の列挙は**箇条書き**で:

```markdown
- **Neovim**: `dot_config/nvim/`（LazyVimベース）
- **Zsh**: `dot_zshrc`, `dot_config/zsh/`
- **Git**: `dot_gitconfig`, `dot_config/git/`
```

カテゴリごとに詳細がある場合は**見出し + 段落**で:

```markdown
### Neovim
LazyVimをベースに `dot_config/nvim/` で管理する。プラグインは...

### Zsh
`dot_zshrc` と `dot_config/zsh/` に分割している。理由は...
```

## 例外

以下のケースでは表を許容する:

- 真に2次元のデータで、すべてのセルが同程度の情報量を持つもの（ベンチマーク結果、互換性マトリクス、APIパラメータ仕様など）
- 比較対象が3つ以上あり、複数の属性を横並びで参照する必要があるもの
- 既存ドキュメントの一貫性を保つために必要な場合

迷ったら箇条書きを選ぶ。表にする明確な理由（2次元構造・横並び比較）が説明できないなら使わない。
