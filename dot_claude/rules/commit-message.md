# コミットメッセージとコミット分割のルール

## 基本原則

- コミットメッセージは変更の「What（何を）」と「Why（なぜ）」を本質的に説明すること
- きっかけ（トリガー）ではなく、変更の内容と理由を書く

## 禁止パターン

以下のような**トリガーをメッセージにしたコミットは禁止**:

- 「レビューコメントに基づき」「指摘を反映」「フィードバック対応」
- 「レビュー対応」「レビューコメント修正」
- 「PR修正」「フィードバック反映」

これらは変更の内容も理由も伝えない。

## レビュー対応時のコミット分割

- レビューコメントへの対応であっても、**各コミットは独立した意味のあるWhy/Whatを持つこと**
- 複数のレビュー指摘に対応する場合、論理的に異なる変更は別々のコミットに分けること
- 各コミットメッセージは「何をなぜ変えたか」を自己完結的に説明し、レビューという文脈がなくても意味が通ること

## 例

良い例:
- `fix(ogas): descriptionからトリガーワード説明を削除し disable-model-invocation との矛盾を解消`
- 2つのレビュー指摘に対し別々のコミット:
  - `fix(auth): トークン検証でexpiry未チェックだった問題を修正`
  - `refactor(auth): ユーザー取得ロジックをrepositoryに移動`

悪い例:
- `fix: レビューコメントに基づくrunbookスキルと手順書の修正`
- `fix: レビューコメント対応` で全指摘をまとめて1コミット

## その他

- **amend + force pushは絶対にしない**。メッセージを間違えても新しいコミットで対応する
