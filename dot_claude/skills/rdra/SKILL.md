# RDRA

RDRA 3.0に基づく要求分析を支援するスキル。RDRA成果物はYAMLフロントマター付きMarkdownで管理し、`traces_to` による単方向トレーサビリティでGOALからユースケースまでのWhy依存チェーンを形成する。

## モード選択

ユーザーの意図に応じてモードを判定する。明示的な指定がなければ、現在の成果物の有無を確認して提案する。

| ユーザーの意図 | モード |
|--------------|--------|
| 要件を整理したい、RDRAで分析したい | 分析 |
| PRDを書きたい、ADRを残したい | 文書生成 |
| 要求をレビューしたい、抜け漏れを確認したい | レビュー |
| 要件が変わった、要求を更新したい | 更新 |

## 出力ディレクトリ構造

```
{project}/
├── rdra/                          # RDRA成果物
│   ├── overview.md                # 全体概観（アクター・ゴール・コンテキスト一覧）
│   ├── contexts/                  # コンテキスト別バーティカルスライス
│   │   └── {context-name}.md      # 4レイヤー縦貫（価値/環境/境界/システム）
│   ├── shared/                    # コンテキスト横断モデル
│   │   ├── information-model.md   # エンティティ定義 + ER図
│   │   └── state-models.md        # 状態モデル定義 + 状態遷移図
│   └── traceability.yaml          # 全要素のトレーサビリティ
├── docs/                          # 生成文書
│   ├── prd.md                     # 要件定義書
│   └── adr/                       # 設計判断記録
│       └── NNN-*.md
└── change-log.md                  # 変更履歴（更新モード）
```

## IDスキーマ

要素種別ベースのセマンティックIDを使い、`traces_to` で上位要素を参照する。

| レイヤー | プレフィックス | 対象 |
|---------|---------------|------|
| システム価値 | `ACTOR-*`, `GOAL-*`, `REQ-*` | アクター、ゴール、要求 |
| 外部環境 | `BIZ-*`, `BUC-*` | コンテキスト、ビジネスUC |
| システム境界 | `UC-*`, `SCR-*`, `EVT-*` | ユースケース、画面、イベント |
| システム | `INFO-*`, `STATE-*`, `COND-*`, `VAR-*` | 情報、状態、条件、バリエーション |

スキーマ詳細は `references/rdra-schemas.md` を参照。

## トレーサビリティ

`traces_to` のみで単方向のWhy依存チェーンを形成する。`traced_from` は保持せず、必要時に逆引きで算出する。

```
GOAL ← REQ ← BUC ← UC
```

全要素を `traceability.yaml` に集約し、孤立要素（GOALに到達しないチェーン）をゼロに保つ。

## モード1: 分析（RDRA Analysis）

ビジネス要件をRDRA 3.0の4レイヤーで構造化する。コンテキスト単位のバーティカルスライスで対話的にヒアリングし、YAML + Mermaid図を生成する。

詳細な進め方は `references/analysis-guide.md` を参照。

### フロー概要

```
Phase 1: スコープ把握 → overview.md ドラフト
Phase 2: コンテキスト特定 → コンテキスト一覧を合意
Phase 3: コンテキスト別深掘り（繰り返し）
  → §価値: ゴール・要求
  → §環境: BUC・業務フロー
  → §境界: UC・画面・イベント
  → §システム: 情報・状態・条件・バリエーションへの参照
  → contexts/{name}.md 生成
Phase 4: 横断モデル整理 → shared/*.md 生成
Phase 5: トレーサビリティ生成 → traceability.yaml
```

### ヒアリング原則

- 1フェーズにつき1テーマに集中する。複数レイヤーを同時に聞かない
- 曖昧な回答には具体例を示して選択肢を提示する
- 各コンテキスト完了時にMermaid図を提示し認識合わせする
- 「他に〜はありますか？」で網羅性を確認する

### Mermaid図

図テンプレートは `references/mermaid-templates.md` を参照。以下の7種を使い分ける:

| RDRAモデル | Mermaid図種 |
|-----------|-------------|
| システムコンテキスト | `graph TB` |
| ビジネスコンテキスト | `graph LR` |
| ビジネスユースケース | `graph TB` |
| 業務フロー | `sequenceDiagram` |
| ロバストネス図 | `flowchart` + `classDef` |
| 情報モデル | `erDiagram` |
| 状態モデル | `stateDiagram-v2` |

YAMLがSSoT。Mermaid図はYAMLから生成されるビューとして扱い、YAML更新時はMermaid図も必ず再生成する。

## モード2: 文書生成（Document Generation）

RDRA成果物を入力とし、PRDとADRを生成する。

### フロー

1. **RDRA成果物の読み込みと検証** — 構造的不備があれば分析モードへの差し戻しを提案
2. **PRD生成** — ゴール・要求→背景と課題、BUC→ユーザーストーリー、UC→機能要件にマッピング。テンプレートは `references/sdd-templates.md` を参照
3. **ADR生成** — 設計判断ポイントをADR候補として提示し、対話で判断理由を明確化

各文書内でRDRA要素IDを参照し、トレーサビリティを維持する。

成果物がない状態で文書生成を求められたら、先に分析モードの実行を提案する。

## モード3: レビュー（RDRA Review）

Agent toolでサブエージェントを起動し、`agents/reviewer.md` の手順に従ってレビューを実行する。メインエージェントのコンテキストから分離することで、生成時のバイアスに影響されにくいレビューを実現する。

### 起動方法

```
Agent tool:
  prompt: |
    agents/reviewer.md を読み込み、その手順に従ってレビューを実行してください。
    RDRA成果物パス: {rdra/の絶対パス}
    レビュー対象: all
```

### 3つのレビュー観点

1. **網羅性** — 4レイヤーの各要素が漏れなく定義されているか
2. **トレーサビリティ** — `traces_to` チェーンがGOALまで到達するか、孤立要素はないか
3. **整合性** — RDRA成果物内の要素間で矛盾がないか、コンテキスト間の依存関係が正しいか

---

## モード4: 更新（RDRA Update）

要件変更時に影響範囲を分析し、関連する成果物を更新する。

### 4 Step フロー

1. **影響分析** — 変更対象の `traces_to` を逆引きし、影響を受ける要素をツリーで可視化
2. **計画提示** — 更新が必要なファイルと変更内容を表形式で提示し、ユーザー承認を得る
3. **更新実行** — RDRA成果物のYAML更新 + Mermaid図再生成、traceability.yaml更新、change-log.md追記
4. **再検証** — レビューモード（トレーサビリティ + 整合性）で更新後の不整合を検出

YAML変更を含む更新では、対象ファイルのMermaid図を必ずYAMLから再生成する。
