# Roadmap Planner

プラットフォームチームのロードマップを「Platform as a Product」の思想で策定・改善するスキル。多様なステークホルダーの要求と技術的制約のバランスを取ったロードマップを生成する。

## 基本思想

プラットフォームチームのロードマップは、プロダクトチームのそれとは本質的に異なる。プロダクトチームはエンドユーザー価値に直結するが、プラットフォームチームの価値は間接的で、複数チーム・複数部門への波及効果として現れる。この違いを理解せずに作ったロードマップは、単なるインフラタスクリストになりがちで、ステークホルダーの支持を得られない。

重要な原則:
- **Output / Outcome / Impact を区別する**。プラットフォームの成果は「何を作ったか（Output）」ではなく「社内顧客の行動がどう変わったか（Outcome）」で測定する。技術的改善指標（ビルド時間、API応答速度、Terraform化率など）はOutputであり、アウトカムではない
- プラットフォームの「社内顧客」は開発者だけではない。セールス、カスタマーサクセス、オペレーションなどビジネスサイド・バックオフィスも価値提供先になりうる。社内顧客が誰かはチームごとに異なるため、Phase 1で明確にする
- **社内顧客の業務負荷の削減**がプラットフォームの存在意義。標準化やコスト削減はその手段
- 社内顧客の声なきロードマップは机上の空論。開発者だけでなくビジネスサイドの声も構造的に拾う
- **最終顧客への価値連鎖を常に意識する**。SaaS企業において、プラットフォームの施策は最終的にプロダクトの顧客（エンドユーザー）への提供価値につながる必要がある。「この施策が社内顧客を通じて最終顧客にどんな価値をもたらすか」を各テーマで明示する
- 60-70%をロードマップ作業に、残りをメンテナンス・サポート・消火活動に充てる現実的な配分を前提とする
- 全リクエスト（自チーム・他チーム依存・緊急）を同じ「予想アウトカム」のレンズで評価する

## モード選択

ユーザーの意図に応じてモードを判定する。明示的な指定がなければ、現在の成果物の有無を確認して提案する。

| ユーザーの意図 | モード |
|--------------|--------|
| ロードマップを新規作成したい、四半期計画を立てたい | 作成 |
| 既存ロードマップを見直したい、優先順位を変えたい | 改善 |
| ロードマップの妥当性を検証したい、抜け漏れを確認したい | レビュー |

## 出力ディレクトリ構造

```
{project}/
├── roadmap/
│   ├── overview.md              # ミッション・戦略・計画期間・成功指標
│   ├── stakeholders.md          # ステークホルダーマップと要求一覧
│   ├── landscape.md             # 外部システム・依存関係マップ
│   ├── tech-debt-registry.md    # 技術的負債レジストリ（3次元リスク評価・テーマブロック関係）
│   ├── themes/                  # テーマ別詳細
│   │   └── {theme-name}.md     # 個別テーマ（背景・施策・依存・タイムライン）
│   ├── timeline.md              # 統合タイムライン（Mermaid Gantt）
│   ├── dependencies.md          # 依存関係の詳細と緩和策
│   └── capacity.md              # キャパシティ配分計画
└── reviews/                     # レビューモード出力
    └── review-report.md
```

## モード1: 作成（Create）

ロードマップをゼロから対話的に構築する。一度に全体を聞き出そうとせず、フェーズごとに1テーマに集中して段階的に深掘りする。

### フロー概要

```
Phase 1: ミッションとスコープ → overview.md ドラフト
Phase 2: ステークホルダー分析 → stakeholders.md
Phase 3: ランドスケープ把握 → landscape.md
Phase 3.5: 技術的負債の棚卸し → tech-debt-registry.md
Phase 4: テーマ策定と優先順位付け → themes/*.md
Phase 5: タイムライン統合 → timeline.md, dependencies.md, capacity.md
```

### Phase 1: ミッションとスコープ

プラットフォームチームの存在意義と計画範囲を明確にする。

#### ヒアリング項目

1. **チームのミッション** — このプラットフォームチームは何のために存在するか
   - 「チームが解決している中核的な課題は何ですか？」
2. **価値提供先** — 誰のためのプラットフォームか
   - 「プラットフォームの主な利用者・受益者は誰ですか？（例: 開発チーム、セールス、CS、オペレーション部門など）」
   - ここで特定した価値提供先がPhase 2以降のステークホルダー分析、Phase 4の優先順位付けの軸になる
3. **計画期間** — どの時間軸で考えるか（四半期／半期／年間）
   - 「今回のロードマップはどの期間をカバーしますか？」
4. **現在の成熟度** — チームとプラットフォームの現在地
   - 「チームの規模と、プラットフォームの現在の利用状況を教えてください」
5. **最終顧客への価値連鎖** — プラットフォームの改善がSaaSの顧客にどうつながるか
   - 「プラットフォームの改善が最終的にプロダクトの顧客にどう届きますか？（例: リリース速度向上→機能提供の早期化、オペレーション効率化→顧客対応品質の向上）」
   - この連鎖はテーマごとの「最終顧客価値」を記述する際の軸になる
6. **成功指標** — 何をもって成功とするか
   - 「このロードマップ期間で達成したい定量的な目標はありますか？（例: オンボーディング時間50%短縮、オペレーション工数30%削減）」

#### 出力

overview.md のドラフトを生成し、ユーザーに確認する。

### Phase 2: ステークホルダー分析

プラットフォームチームは社内外に多種多様なステークホルダーを持つ。誰の声を聞き、誰に報告し、誰と協働するかを構造化することが、現実的なロードマップの前提条件になる。

詳細な進め方は `references/stakeholder-guide.md` を参照。

#### ヒアリングの流れ

1. **ステークホルダーの列挙** — 「プラットフォームに関わる人・チーム・外部組織をすべて挙げてください」
2. **分類** — 各ステークホルダーを4象限（高影響力×高関心 / 高影響力×低関心 / 低影響力×高関心 / 低影響力×低関心）にマッピング
3. **要求の収集** — 各ステークホルダーがプラットフォームに求めていること
4. **関わりの実態の深掘り** — ステークホルダーごとに以下を確認する:
   - 「このステークホルダーに対して、あなたのチームは**何を期待し、何を期待していないか**」
   - 「このステークホルダーの業務のうち、**自チームで完結する領域**はどこか」
   - 「名目上の役割と**実際の運用実態**にギャップはないか」
5. **コンフリクトの特定** — ステークホルダー間で矛盾する要求

#### 出力

stakeholders.md を生成し、Mermaid `quadrantChart` でステークホルダーマップを可視化する。

### Phase 3: ランドスケープ把握

プラットフォームが関わる外部システム、依存サービス、技術スタックの全体像を把握する。

#### ヒアリング項目

1. **外部システム連携** — 「プラットフォームが連携している外部システム・サービスを教えてください」
2. **内部依存** — 「他チームが提供するサービスで、プラットフォームが依存しているものはありますか？」
3. **技術的制約** — 「技術選定上の制約やマイグレーション計画はありますか？」
4. **技術的負債** — 「放置すると将来的にコストが増大する技術的負債はありますか？」

#### 出力

landscape.md を生成し、Mermaid `graph` でシステム依存関係図を可視化する。

### Phase 3.5: 技術的負債の棚卸し

Phase 3で把握した技術的負債を独立したレジストリとして構造化し、リスクを多面的に評価する。

#### 進め方

1. **負債の確認と追加ヒアリング** — Phase 3で挙がった技術的負債を確認し、追加の負債がないかヒアリングする
   - 「Phase 3で挙がった負債以外に、放置すると将来的にテーマの実行を阻害しうる技術的負債はありますか？」
2. **3次元リスク評価** — 各負債について以下の3次元でリスクを評価する（各1-5）
   - プロダクト影響: 顧客体験・プロダクトKPIへの影響
   - 開発速度影響: チーム速度・リードタイムへの影響
   - セキュリティ/信頼性: インシデントリスク・コンプライアンス違反
   - **総合リスクスコアは3次元の最大値**とする（加重平均ではなく最悪ケースで判断。セキュリティリスク5が他次元で希釈されるのを防止）
3. **テーマへのブロック関係の特定** — 各負債が解消されないと着手・完了できないテーマを特定する（Phase 4完了後に再確認）
4. **遅延コスト評価** — 負債ごとに遅延コストタイプ（時間感応/線形減衰/時間非依存）を評価する

#### 出力

tech-debt-registry.md を生成し、ユーザーに確認する。ブロック関係はPhase 4完了後に最終確認する。

### Phase 4: テーマ策定と優先順位付け

Phase 1-3の情報を統合し、ロードマップのテーマ（大きな取り組みの単位）を策定する。

詳細な優先順位付け手法は `references/prioritization-guide.md` を参照。

#### 進め方

1. **テーマ候補の抽出** — ステークホルダー要求、tech-debt-registry.md のリスク評価結果、戦略目標からテーマ候補を洗い出す。総合リスクスコア4-5の負債はテーマ化を検討する
2. **テーマの粒度検証** — 各テーマ候補について以下のヒューリスティックで分解の必要性を判定する:
   - 「このテーマ内で、**異なるタイミングで着手・完了できる部分**はないか」
   - 「このテーマ内に、**異なる依存関係を持つサブテーマ**が含まれていないか」
   - 「このテーマの完了を説明するとき、**複数の独立した技術的関心事**に言及する必要がないか」
   該当する場合はテーマを分解し、それぞれに独自の依存関係とタイムラインを設定する。ただし、分解後のテーマが単独では戦略的意味を持たない粒度まで細分化しない（分解はロードマップ上で独立にスケジュール可能な単位を目安とする）
3. **テーマの構造化** — 各テーマについて以下を明確に分離して整理する:
   - **開発テーマ（What）**: 具体的に何を作る・変更するか（技術的な先行関係はここに記述）
   - **アウトカム（Outcome）**: 「誰が」「何をできるようになるか」を社内顧客の行動変化として記述する（テーマとは同列に扱わず、テーマから導かれる結果として記述）
   - **背景（Why）**: Why this（なぜこのテーマか）は必須。Why now（なぜ今か）、Why us（なぜ自チームか）は自明でない場合のみ記述する
   - 依存関係も併せて整理する
   - **アウトカム品質チェック** — 各テーマのアウトカム記述時に以下を確認する:
     - [ ] 「誰が」に具体的なステークホルダー名が入っているか
     - [ ] 「何ができるようになるか」が行動変化を動詞で表しているか
     - [ ] 技術的指標（ビルド時間、応答速度、カバレッジ率など）がアウトカムに混入していないか（それらはOutputとして施策セクションに記述する）
4. **優先順位付け** — Leverage × Velocity の2×2マトリクスで優先度を決定
   - **Leverage（価値増幅度）**: 社内顧客の行動変化が最終顧客の価値にどの程度直結するか
   - **Velocity（行動変化の深さ）**: 社内顧客が新しいケイパビリティを獲得するか、既存業務の効率化に留まるか
5. **Cost of Delay 評価** — 着手を遅らせた場合に失われる価値を評価し、順序を調整
6. **ユーザーとの合意** — 優先順位付け結果を提示し、フィードバックを反映
7. **負債→テーマ紐付けの最終確認** — Phase 3.5で仮設定したブロック関係を、確定したテーマに基づいて最終確認する。tech-debt-registry.md の `blocks` とテーマの `blocking_debts` が双方向で一致することを検証する

テーマ分類の指針:
- **Golden Path**: 利用者が最も頻繁に踏む経路の標準化・自動化（開発者のデプロイフロー、オペレーションの定型業務など）
- **Foundation**: インフラ基盤、セキュリティ、可観測性などの土台
- **User Experience**: 利用者体験の改善（ツーリング、ドキュメント、オンボーディング。開発者向けに限らない）
- **Tech Debt**: 計画的な負債返済
- **Support & Operations**: 運用負荷の削減、セルフサービス化

#### 出力

各テーマごとに themes/{theme-name}.md を生成する。

### Phase 5: タイムライン統合

全テーマを時間軸上に配置し、実行可能な計画に統合する。

#### 進め方

1. **依存関係の解決** — テーマ間・外部依存を整理し、実行順序を決定
2. **キャパシティ配分** — 60-70%ロードマップ / 20-25%メンテナンス / 10-15%バッファの配分を設定
3. **マイルストーン設定** — 各テーマの開始・完了時期とチェックポイントを配置
4. **リスク識別** — 外部依存や不確実性の高いテーマにリスクフラグを付与

#### 出力

- timeline.md — Mermaid `gantt` チャートで統合タイムラインを可視化
- dependencies.md — 依存関係の詳細と緩和策
- capacity.md — キャパシティ配分計画

## モード2: 改善（Improve）

既存ロードマップを分析し、課題を特定して改善案を提示する。

### フロー概要

```
Step 1: 現状把握 → 既存成果物を読み込み
Step 2: ギャップ分析 → 問題点の構造化
Step 3: 改善提案 → 具体的な変更案の提示
Step 4: 成果物更新 → 合意した変更を反映
```

### Step 1: 現状把握

既存のロードマップ成果物を読み込み、以下を確認する:

- ミッションと成功指標は明確か
- ステークホルダーの要求がテーマに反映されているか
- 依存関係が管理されているか
- キャパシティ配分は現実的か

### Step 2: ギャップ分析

以下の観点で問題を洗い出す:

| 観点 | チェック内容 |
|------|------------|
| ステークホルダーカバレッジ | 要求未反映のステークホルダーはいないか |
| 依存リスク | 外部依存の緩和策が不十分なテーマはないか |
| バランス | Golden Path / Foundation / DX / Tech Debt の配分は適切か |
| 実現可能性 | キャパシティに対してテーマが過多でないか |
| 指標との整合 | テーマが成功指標の達成に寄与するか |

### Step 3: 改善提案

問題ごとに具体的な改善案を提示する。変更のインパクト（影響範囲と工数）を見積もり、ユーザーと合意してから適用する。

### Step 4: 成果物更新

合意した変更を既存の成果物に反映する。変更履歴をoverview.mdの末尾に追記する。

## モード3: レビュー（Review）

Agent toolでサブエージェントを起動し、`agents/reviewer.md` の手順に従ってレビューを実行する。メインエージェントのコンテキストから分離することで、作成時のバイアスに影響されにくいレビューを実現する。

### 起動方法

```
Agent tool:
  prompt: |
    agents/reviewer.md を読み込み、その手順に従ってレビューを実行してください。
    ロードマップ成果物パス: {roadmap/の絶対パス}
    レビュー対象: all
    参照ドキュメント: {関連ドキュメントのパス（任意、複数可）}
```

ユーザーがロードマップ外の関連ドキュメント（仕様書、ADR、PRD等）を持っている場合は、参照ドキュメントとして渡す。これによりレビューエージェントがドメインコンテキストを把握し、誤った指摘を減らせる。

### 5つのレビュー観点

1. **ステークホルダーカバレッジ** — 重要なステークホルダーの要求がテーマに反映されているか
2. **依存関係の健全性** — 循環依存、緩和策なしの外部依存、クリティカルパス上のリスク
3. **バランスと持続可能性** — テーマ分類の偏り、キャパシティの過負荷、バッファ不足
4. **実現可能性** — タイムラインの妥当性、リソース制約との整合
5. **戦略整合性** — ミッション・成功指標とテーマの対応関係

## ヒアリング原則

- 1フェーズにつき1テーマに集中する。複数フェーズを同時に聞かない
- 曖昧な回答には具体例を示して選択肢を提示する
- 各フェーズ完了時にMermaid図を提示し認識合わせする
- 「他に〜はありますか？」で網羅性を確認する
- ステークホルダーの要求が矛盾する場合はトレードオフを明示し、判断をユーザーに委ねる
