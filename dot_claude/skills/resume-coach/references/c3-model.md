# C3モデル（シーキューブモデル）詳細リファレンス

STAR法の構造的限界を補うために設計された、職務経歴書作成フレームワーク。

出典: https://qiita.com/newta/items/248390917376fc96e51d

## モデルの全体像

3つの視点（縦軸）× 3つの時間軸（横軸）= 9セルのマトリックス。

```
              │  課題          │  役割・行動      │  結果
─────────────┼───────────────┼────────────────┼──────────────
Context      │ 環境の課題     │ 環境の動き       │ 環境の変化
（周囲の客観）│               │                  │
─────────────┼───────────────┼────────────────┼──────────────
Capacity     │ 任された課題   │ 公式な役割       │ 達成した責務
（自分の客観）│               │                  │
─────────────┼───────────────┼────────────────┼──────────────
Choice       │ 見立てた課題   │ 判断と選択       │ 判断の帰結
（自分の主観）│               │                  │
```

## 各層の詳細

### Context（周囲の客観）— 「自分がいなくても成り立つ環境」

自分の存在を消して描写する。解釈や評価を一切入れない。

**書くべきもの:**
- チームの人数・構成（「5名チーム、うちバックエンド3名」）
- 事業の状況（「月間アクティブユーザー50万人」「シリーズBの資金調達直後」）
- 技術的な制約（「レガシーなモノリスアーキテクチャ」「本番稼働中で停止不可」）
- 時間的な制約（「3ヶ月以内のリリースが必須」）
- 組織的な状況（「事業部間のデータ連携に統一基盤がなかった」）

**よくあるNG:**
- NG: 「チームの技術力が低く、私がリードする必要があった」→ 解釈が入っている
- OK: 「チーム5名、うち3名が入社半年以内」→ 事実のみ

### Capacity（自分の客観）— 「何を任されていたか」

第三者が見て確認できる、公式な役割と責務。

**書くべきもの:**
- 肩書き・ロール（「テックリード」「バックエンド担当」）
- 担当スコープ（「認証基盤の設計・実装」「API設計からデプロイまで」）
- 他メンバーとの関係（「フロントエンド2名と協働」「PdMと要件定義を並行」）
- 報告ライン（「CTOに直接報告」）

**よくあるNG:**
- NG: 「最適なアーキテクチャを選定し、チームを正しい方向に導いた」→ 判断が入っている
- OK: 「アーキテクチャ選定とチームへの技術方針策定を担当」→ 事実のみ

### Choice（自分の主観）— 「頭の中で起きていたこと」

ここだけに思考・判断・トレードオフを記載する。この層が職務経歴書の差別化ポイント。

**書くべきもの:**
- 判断の根拠（「将来の拡張性を重視し、マイクロサービス化ではなくモジュラーモノリスを選択」）
- 比較した選択肢（「Auth0とAuthleteを比較検討し、カスタマイズ性を優先してAuthleteを採用」）
- トレードオフの評価（「開発速度を犠牲にして型安全性を優先した。短期的にはコストだが、チーム拡大時のバグ率低減を見込んだ」）
- 想定外の事態への対応（「当初の設計ではスケーラビリティが不足することが判明し、〜に方針転換」）
- 振り返りの知見（「結果的に〜は正解だったが、〜はもっと早く着手すべきだった」）

## 活用のポイント

### 分析ツールとして使う

9セルすべてを職務経歴書に書く必要はない。経験を整理・分析するためのフレームワークとして使い、最終的な経歴書には自然な形で織り込む。

### 層の混同を防ぐ

最もよくある失敗は、Context層に主観的な解釈を入れること、Capacity層に判断を入れること。各層に記載すべきものを厳密に守ることで、「事実」と「判断」が明確に分離され、読み手が候補者の思考力を正確に評価できるようになる。

### 「指示待ち作業者」テスト

Choice層が薄いと、どんなに立派な経歴でも「言われたことを実装した人」に見える。逆に、Choice層が充実していれば、「構造に責任を持って判断した人」として伝わる。
