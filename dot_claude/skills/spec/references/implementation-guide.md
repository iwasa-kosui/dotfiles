# 実装生成モード詳細ガイド

仕様書を入力とし、タスクリスト・実装コード・ADR・受け入れテストを生成する。SDDの「Specify → Plan → Tasks → Implement」ワークフローをRDRAトレーサビリティ付きで実行する。

## 前提条件

- RDRA成果物（`rdra/`）が存在すること
- SDD文書（`specs/prd.md`, `specs/spec.md`）が存在すること
- `traceability.yaml` が存在し、GOAL → SPEC のチェーンが構築済みであること

不足がある場合は、仕様作成モード（または分析モード）への差し戻しを提案する。

---

## Step 1: 技術スタック・アーキテクチャのヒアリング

### ヒアリング項目

1. **言語・フレームワーク**
   - 「使用する言語とフレームワークを教えてください」
   - 既存プロジェクトであれば `package.json`, `go.mod`, `Cargo.toml` 等を読み取る

2. **アーキテクチャ方針**
   - 「モノリス/マイクロサービス、レイヤードアーキテクチャ/クリーンアーキテクチャ等の方針はありますか？」

3. **インフラ・デプロイ**
   - 「デプロイ先（AWS/GCP/自前等）やCI/CDの構成を教えてください」

4. **チームの慣習**
   - 「コーディング規約、テストフレームワーク、ディレクトリ構成のルールはありますか？」

### ADR候補の自動検出

ヒアリング中に検出された技術的判断ポイントをADR候補として記録する。Step 3で正式にADRを生成する。

---

## Step 2: タスク分解

### 分解の進め方

1. `specs/spec.md` の各仕様項目（SPEC-*）を読み込む
2. 各仕様項目を実装可能なタスク単位に分解する
3. タスク間の依存関係を特定する
4. `specs/tasks.md` を生成する

### タスクの粒度

- 1タスク = 1つのPull Requestでレビュー可能な単位
- 目安: 1タスクあたり100-300行程度の変更
- データモデル → API → UI の順で依存を意識する

### 各タスクに付与する情報

```markdown
### TASK-NNN: {タスク名}
- **traces_to**: SPEC-*, UC-*, FR-*
- **依存**: TASK-* （なければ「なし」）
- **受け入れ条件**:
  - {BUC/UCの記述から導出した条件}
- **成果物**: {生成するファイルパス}
```

### 受け入れ条件の導出

受け入れ条件はRDRA成果物から導出する:

| RDRA要素 | 導出する受け入れ条件 |
|---------|-------------------|
| BUC | ビジネスフローが正常に完了すること |
| UC | ユーザーが操作を完了できること |
| COND-* | 条件を満たす場合/満たさない場合の動作 |
| VAR-* | 各バリエーションでの動作 |
| STATE-* | 状態遷移が正しく行われること |

---

## Step 3: 実装・ADR・テスト生成

タスクリストのユーザー承認後、タスク単位で実装を進める。

### 実装コードの生成

- Step 1でヒアリングした技術スタック・アーキテクチャに従う
- 各ファイルにコメントで `traces_to: TASK-NNN` を記載する
- 生成コードはあくまで「たたき台」。人間がレビューする前提で、完璧を目指さず意図が伝わるコードを書く

### ADRの生成

実装中に技術的判断が必要になった場合、ADRを生成する:

- Step 1で検出したADR候補を正式にADRとして記録
- 実装中に新たに発生した判断もADRとして記録
- `specs/adr/NNN-*.md` に保存

### 受け入れテストの生成

RDRA BUC/UCから受け入れテストを生成する:

1. 各BUCの業務フローを正常系テストケースに変換
2. 条件（COND-*）を境界値テストケースに変換
3. バリエーション（VAR-*）を組み合わせテストケースに変換
4. 状態遷移（STATE-*）を状態遷移テストケースに変換
5. テストフレームワークに合わせたテストコードを生成

テストコードにもコメントで `traces_to: BUC-NNN` / `UC-NNN` を記載する。

---

## Step 4: トレーサビリティ更新・検証

### traceability.yaml の更新

生成したタスク・コード・テストの要素を追記する:

```yaml
- id: "TASK-001"
  type: task
  name: "タスク名"
  defined_in: "specs/tasks.md"
  traces_to: ["SPEC-001", "UC-001"]
```

### チェーン完結の検証

以下を検証し、問題があればユーザーに報告する:

1. すべてのTASKが `traces_to` でSPEC/UC/FRを参照しているか
2. GOAL → REQ → BUC → UC → FR → SPEC → TASK のチェーンが途切れていないか
3. Specのうちタスク化されていない項目がないか（カバレッジ）

### カバレッジレポートの出力

```
## タスクカバレッジ

| メトリクス | 値 |
|-----------|-----|
| 総Spec項目数 | N |
| タスク化済み | M |
| 未タスク化 | N-M |
| カバレッジ | M/N * 100% |

### 未タスク化の仕様項目
- SPEC-003: ...
- SPEC-007: ...
```
